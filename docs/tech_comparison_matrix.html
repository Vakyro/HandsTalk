Comparativa de Opciones Tecnológicas

Frontend Framework

Opción Ventajas Desventajas Recomendación
Next.js 15 ✓ SSR/SSG nativo, API routes, Vercel integration, TypeScript first Curva de aprendizaje moderada ELEGIDO
Vite + React Más ligero, bundling rápido No tiene API routes, SPA only Alternativa si no necesitas backend
Nuxt 3 Bueno para Vue, SSR similar a Next Comunidad React mucho más grande para ML No recomendado para este proyecto

Database

Opción Ventajas Desventajas Recomendación
Neon (PostgreSQL) ✓ Serverless, autoscale, cheap, SQL completo Cold starts ocasionales ELEGIDO
Firebase / Firestore NoSQL, tiempo real, easy auth Queries complejas difíciles, costos por lectura, vendor lock-in Alternativa si prefieres NoSQL
Supabase (PostgreSQL) PostgreSQL + auth + storage integrados Pricing puede ser caro a escala Alternativa viable (usamos su Storage)

ML Framework (Entrenamiento)

Opción Ventajas Desventajas Recomendación
PyTorch + PyTorch Lightning ✓ Flexible, research-friendly, comunidad gigante, bueno para Transformers Más verbose que TF/Keras ELEGIDO
TensorFlow + Keras Bueno para production, mais alto nivel Menos flexible para investigación, comunidad ML menos activa Alternativa si tienes experiencia TF
JAX Muy flexible, functional Curva de aprendizaje empinada, menos ejemplos de seq2seq Solo si eres un expert en JAX

Feature Extraction (MediaPipe)

Opción Ventajas Desventajas Recomendación
MediaPipe (Google) ✓ Real-time, browser + server, multi-platform, free, excelente para señas No perfecta (puede perder dedos en oclusión) ELEGIDO
OpenPose Muy preciso para cuerpo Lento, no tiene hands good, menos desarrollado ahora Complemento opcional para refinar
Blazepose Ligero, mobile-friendly Menos preciso que MediaPipe Pose completo Alternativa si necesitas móvil offline

Temporal Encoding (Sequence Modeling)

Opción Ventajas Desventajas Recomendación
Transformer Encoder ✓ Self-attention (mejor coarticulación), paralelizable, visualizable, estado-del-arte Necesita más datos, compute costoso ELEGIDO
TCN (Temporal Conv) Más ligero, bueno para secuencias cortas Receptive field limitado, menos interpretable Alternativa para MVP ligero
LSTM Bidireccional Clásico, probado, menos compute Vanishing gradient, peor coarticulación que attention Fallback si Transformer es muy lento

Translation (Gloss → Español)

Opción Ventajas Desventajas Recomendación
HuggingFace Transformers (fine-tuned NMT) ✓ Modelos pre-entrenados, easy to fine-tune, comunidad gigante Requiere tuning, dataset LSM→español para entrenar ELEGIDO
Claude API (LLM) Muy bueno para correcciones y contexto, fácil integración Costo por token, no es modelo entrenado para señas, latencia Complemento (v2.5+), no primario
Google Translate API Confiable, mantenido por Google No entiende glosses de señas, costo por request, no customizable No recomendado para este caso

Authentication

Opción Ventajas Desventajas Recomendación
NextAuth.js v5 ✓ OAuth2, JWT nativo Next.js, sin vendor lock-in, gratuito Setup inicial, mantenimiento propio ELEGIDO
Auth0 Enterprise-ready, muchas features, soporte 24/7 Caro, overkill para MVP, vendor lock-in Alternativa para v3.0 (enterprise)
Supabase Auth Integrado con Supabase, fácil si usas Storage Menos flexible que NextAuth Alternativa si eliges Supabase para todo

Video Storage

Opción Ventajas Desventajas Recomendación
Supabase Storage ✓ Integrado, fácil auth, CDN, barato para MVP Escalabilidad limitada a muy grande escala ELEGIDO (MVP)
AWS S3 Escalable infinitamente, CloudFront CDN, barato a escala Setup más complejo, curva aprendizaje AWS, pricing opaco Escala a v2.5+ si creces exponencialmente
Cloudinary Optimización automática, transformación de video, fácil Menos control, pricing variable, vendor lock-in Alternativa si necesitas transformación automática

Deployment / Hosting

Opción Ventajas Desventajas Recomendación
Vercel ✓ Next.js nativo, serverless, auto-scale, CDN, free tier, preview deployments Vendor lock-in, procesamiento ML limitado en functions ELEGIDO (Frontend + API)
Railway / Render Flexible, mejor para monolitos, pricing simple No optimizado para Next.js, más overhead Alternativa si necesitas control total
Docker + Kubernetes Control total, multi-region, bueno para ML (GPU) Overhead operacional alto, overkill para MVP v3.0+ cuando necesites auto-scaling ML

Job Queue (Async Processing)

Opción Ventajas Desventajas Recomendación
Bull (Redis) ✓ Fácil, in-memory, persistence, retry logic, delay jobs Single point of failure (Redis), no distributed state ELEGIDO (MVP)
Temporal.io Distribuido, resiliente, logs de ejecución, muy robusto Setup más complejo, overhead operacional, overkill para MVP Upgrade cuando escales a v2.5+
AWS SQS Managed, auto-scale, integración con Lambda Vendor lock-in AWS, pricing por mensaje Alternativa si usas AWS para todo

📊 Resumen de Decisiones

  • Frontend: Next.js 15 (SSR, API routes, Vercel native)
  • Database: Neon PostgreSQL (serverless, autoscale, SQL)
  • ML Training: PyTorch + Lightning (flexible, Transformers)
  • Features: MediaPipe (real-time, browser + server)
  • Temporal: Transformer Encoder (coarticulación, attention)
  • Translation: HuggingFace NMT (fine-tuned, comunidad)
  • Auth: NextAuth.js (sin vendor lock-in, OAuth2)
  • Storage: Supabase (MVP), AWS S3 (scale v2.5+)
  • Deployment: Vercel (Next.js native)
  • Queue: Bull + Redis (simple, effective)
